# Neil Agent 可视化开发计划

本文记录 `/cockpit` 基础驾驶舱之后的高级可视化路线。它与主线功能计划分开维护，目标是允许按小批次、低耦合地偶尔推进，而不要求一次完成完整 TUI。

## 当前基线

已完成的 `/cockpit` 是一个只读 Rich 快照：

- 任务矩阵：当前计划步骤和最近一次质量检查。
- 基础上下文断层图：字符/token 软预算、历史裁剪和最近服务端 `usage`。
- 基础安全盾：直接执行/逐次审批工具数量、工作区边界、审计和 OS 沙箱现状。
- 工作区信号：项目指令元数据、进程内文件检查点和有界 Git 状态。

该命令不调用模型、不修改 Agent 状态、不写文件、不启动额外事件循环，也不显示 prompt、thinking、工具参数/结果正文、API Key 或 `AGENTS.md` 正文。

Phase 0A 事件基础也已完成：

- 不可变、版本化的 `RuntimeEvent`，使用唯一事件 ID、稳定关联 ID、父事件 ID 和严格的阶段/状态契约。
- Agent 回合、模型请求、工具调用、审批和质量检查已形成元数据专用事件链。
- 统一字段白名单拒绝未知字段、控制字符、越界文本和负数计数，不允许 prompt、工具参数/结果或审批预览正文进入事件。
- `EventBus` 为每个观察者分配独立有界队列；发布不等待观察者，回调异常和队列溢出只进入统计。
- 事件总线仍是可选的进程内旁路；现有 CLI 和 `/cockpit` 默认不创建它，也不改变任何输出协议。

## 架构原则

1. **先数据、后界面**：事件模型、存储和投影必须能在纯文本测试中独立验证，Textual 只消费稳定投影。
2. **只读旁路**：观察者超时、崩溃、队列溢出或界面关闭都不能改变 Agent 执行结果。
3. **事件而非回调拼图**：生命周期 hooks 用于宿主扩展和策略控制；可视化事件只描述已经发生的事实，两者不混用。
4. **图是投影，不是真相源**：DAG、时间线、统计和热力图都从同一事件流派生，不能各自维护隐式状态。
5. **元数据优先**：默认记录名称、阶段、耗时、规模、状态和稳定 ID；敏感正文保持零记录。
6. **保持现有协议**：不能改变交互 CLI、`text`、`json` 或 `stream-json` 的既有输出契约。

建议的数据流：

```text
Agent / ToolRegistry / Context
            │
            ▼
       RuntimeEvent
            │
            ▼
   bounded EventBus ──► optional JSONL EventStore
            │
            ├──► ExecutionGraphProjector
            ├──► TimelineProjector
            └──► MetricsProjector
                       │
                       ▼
               Textual / Rich views
```

## 分阶段路线

| 阶段 | 状态 | 范围 | 独立交付物 |
| --- | --- | --- | --- |
| 基础驾驶舱 | 已完成 | Rich 只读运行时快照 | `/cockpit` |
| Phase 0A | 已完成 | `RuntimeEvent`、稳定 ID、脱敏和有界 `EventBus` | 内存事件流与单元测试 |
| Phase 0B | 待开发 | 可选 JSONL 事件存储、确定性 DAG/时间线投影 | 文本回放命令或测试夹具 |
| Phase 1 | 待开发 | Textual 全屏骨架、实时 DAG、token/耗时统计 | `/cockpit --live` 或独立入口 |
| Phase 2A | 待开发 | Context Tomography 高级版 | 上下文来源、轮次和裁剪可视化 |
| Phase 2B | 待开发 | Security Shield 高级版 | 权限色带、审批流和边界状态 |
| Phase 3A | 待开发 | Time Machine 只读回放 | 快照时间轴和历史状态浏览 |
| Phase 3B | 延后 | 安全恢复 | 经审批、并发校验和失败原子性的恢复 |
| Phase 4 | 可选 | Neural Map | 仓库文件活动热力图 |

完整高级路线预计约 7–11 周的连续开发量；按下面的小批次推进时，每次可以只完成一个可测试切片。

## 可随手推进的检查清单

### Phase 0A：事件基础

- [x] 定义不可变 `RuntimeEvent`，包含版本、事件 ID、父事件 ID、时间、阶段、状态和有界元数据。
- [x] 为一次 Agent 回合、模型请求、工具调用、审批和质量检查分配稳定关联 ID。
- [x] 建立字段白名单与统一脱敏器，拒绝未知字段和控制字符。
- [x] 实现有界内存 `EventBus`；慢观察者隔离，队列满时只丢观察数据并报告计数。
- [x] 验证无观察者、有故障观察者和队列溢出时，Agent 结果与当前实现一致。

### Phase 0B：存储与投影

- [ ] 增加显式开启的版本化 JSONL `EventStore`，使用大小上限和安全轮转。
- [ ] 从事件生成 `ExecutionGraph`，而不是在 Agent 内维护第二份图状态。
- [ ] 定义节点状态机和缺失/重复/乱序事件的确定性处理规则。
- [ ] 提供不依赖 Textual 的纯文本回放或固定夹具。
- [ ] 用 10,000 个合成事件验证内存、耗时和稳定性。

### Phase 1：实时 DAG

- [ ] 引入 Textual 前先确定 Python 3.13 兼容版本并记录依赖决策。
- [ ] 实现全屏应用的启动、退出、终端 resize 和异常降级。
- [ ] 展示 Agent 回合 → 模型请求 → 工具调用 → 检查的实时 DAG。
- [ ] 增加节点筛选、详情抽屉和耗时/token 汇总；详情仍只显示安全元数据。
- [ ] 保留现有 `/cockpit`，在非交互终端和 Textual 不可用时继续作为降级入口。

### Phase 2A：Context Tomography

- [ ] 分层展示系统固定开销、工具 schema、项目指令、历史轮次和当前链路。
- [ ] 区分本地估算与服务端实测，不把历史 `usage` 预测成下次费用。
- [ ] 标出被裁剪轮次、压缩检查点和高占用工具结果，但不展示正文。
- [ ] 提供预算阈值预警和“如果增加 N 字符”的本地模拟，不触发模型请求。

### Phase 2B：Security Shield

- [ ] 用一致色带展示直接执行、逐次审批、永久禁止和不可用能力。
- [ ] 把审批请求关联到对应 DAG 节点，并显示预览绑定是否仍有效。
- [ ] 明确区分应用层白名单与 OS 沙箱，绝不把前者标成后者。
- [ ] 展示路径、网络、命令和审计边界的状态变化；敏感值保持隐藏。

### Phase 3：Time Machine

- [ ] 先完成只读时间轴：浏览事件、会话分支、压缩和文件检查点。
- [ ] 回放只重建投影，不重新调用模型或工具。
- [ ] 将“查看历史”与“恢复状态”设计成两个独立能力。
- [ ] 恢复必须具有明确审批、哈希/版本并发校验、原子应用和失败回滚。
- [ ] 多文件可靠恢复完成前，继续把 Git 作为主回退机制。

### Phase 4：Neural Map

- [ ] 从已脱敏文件活动事件聚合，不扫描或保存文件正文。
- [ ] 提供目录聚类、读/写/检查热度、时间窗口和风险着色。
- [ ] 大仓库采用增量布局和节点上限，超限时聚合到目录级。
- [ ] 先用静态夹具验证信息价值，再决定是否进入实时主界面。

## 每个批次的完成标准

- 新增数据结构有严格版本和边界测试。
- 运行时值按纯文本渲染，窄终端或 resize 不破坏布局。
- 不记录 prompt、thinking、工具参数值、工具结果正文、API Key 或项目指令正文。
- 不改变无可视化观察者时的 Agent 行为、错误分类和结构化输出。
- Ruff、格式检查、mypy、pytest 和离线评测全部通过。
- 在 `DevelopmentRecords.md` 记录本批次的实际交付、限制和下一小步。
